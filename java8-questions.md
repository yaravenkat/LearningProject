<p>What is the structure and features of using a lambda expression?
A lambda is a set of instructions that can be separated into a separate variable and then repeatedly called in various places of the program.</p>
<p>The basis of the lambda expression is the <em>lambda operator</em> , which represents the arrow <code>-&gt;</code>. This operator divides the lambda expression into two parts: the left side contains a list of expression parameters, and the right actually represents the body of the lambda expression, where all actions are performed.</p>
<p>The lambda expression is not executed by itself, but forms the implementation of the method defined in the functional interface. It is important that the functional interface should contain only one single method without implementation.</p>

What is Stream?</h2>
<p>An interface <code>java.util.Stream</code> is a sequence of elements on which various operations can be performed.</p>
<p>Operations on streams can be either intermediate (intermediate) or final (terminal) . Final operations return a result of a certain type, and intermediate operations return the same stream. Thus, you can build chains of several operations on the same stream.</p>
<p>A stream can have any number of calls to intermediate operations and the last call to the final operation. At the same time, all intermediate operations are performed lazily and until the final operation is called, no actions actually happen (similar to creating an object <code>Thread</code>or <code>Runnable</code>, without a call <code>start()</code>).</p>
<p>Streams are created based on sources of some, for example, classes from <code>java.util.Collection</code>.</p>
<p>Associative arrays (maps), for example <code>HashMap</code>, are not supported.</p>
<p>Operations on streams can be performed both sequentially and in parallel.</p>
<p>Streams cannot be reused. As soon as some final operation has been called, the flow is closed.</p>
<p>In addition to the universal object, there are special types of streams to work with primitive data types <code>int</code>, <code>long</code>and <code>double</code>: <code>IntStream</code>, <code>LongStream</code>and <code>DoubleStream</code>. These primitive streams work just like regular object streams, but with the following differences:</p>
<ul>
<li>use specialized lambda expressions, for example, <code>IntFunction</code>or <code>IntPredicate</code>instead of <code>Function</code>and <code>Predicate</code>;</li>
<li>support additional end operations <code>sum()</code>, <code>average()</code>, <code>mapToObj()</code>.</li>
</ul>

Tell us about parallel processing in Java 8?</h2>
<p>Streams can be sequential and parallel. Operations on sequential streams are performed in one processor thread, on parallel streams - using several processor threads. Parallel streams use the shared stream <code>ForkJoinPool</code>through the static <code>ForkJoinPool.commonPool()</code>method. In this case, if the environment is not multi-core, then the stream will be executed as sequential. In fact, the use of parallel streams is reduced to the fact that the data in the streams will be divided into parts, each part is processed on a separate processor core, and in the end these parts are connected, and final operations are performed on them.</p>
<p>You can also use the <code>parallelStream()</code>interface method to create a parallel stream from the collection <code>Collection</code>.</p>
<p>To make a regular sequential stream parallel, you must call the <code>Stream</code>method on the object <code>parallel()</code>. The method <code>isParallel()</code>allows you to find out if the stream is parallel.</p>
<p>Using, methods <code>parallel()</code>and <code>sequential()</code>it is possible to determine which operations can be parallel, and which only sequential. You can also make a parallel stream from any sequential stream and vice versa:</p>
<div class="highlight highlight-source-java"><pre>collection
  .stream ()
  .peek ( <span class="pl-c1">...</span> ) <span class="pl-c"><span class="pl-c">//</span> operation is sequential</span>
  .parallel ()
  .map ( <span class="pl-c1">...</span> ) <span class="pl-c"><span class="pl-c">//</span> the operation can be performed in parallel,</span>
  .sequential ()
  .reduce ( <span class="pl-c1">...</span> ) <span class="pl-c"><span class="pl-c">//</span> operation is sequential again</span></pre></div>
<p>As a rule, elements are transferred to the stream in the same order in which they are defined in the data source. When working with parallel streams, the system preserves the sequence of elements. An exception is a method <code>forEach()</code>that can output elements in random order. And in order to maintain the order, it is necessary to apply the method <code>forEachOrdered()</code>.</p>
<ul>
<li>Criteria that may affect performance in parallel streams:</li>
<li>Data size - the more data, the more difficult it is to separate the data first, and then combine them.</li>
<li>The number of processor cores. Theoretically, the more cores in a computer, the faster the program will work. If the machine has one core, it makes no sense to use parallel threads.</li>
<li>The simpler the data structure the stream works with, the faster operations will occur. For example, data from is <code>ArrayList</code>easy to use, since the structure of this collection assumes a sequence of unrelated data. But a type collection  <code>LinkedList</code>is not the best option, since in a sequential list all the elements are connected with previous / next. And such  data is difficult to parallelize.</li>
<li>Operations with primitive types will be faster than with class objects.</li>
<li>It is highly recommended that you do not use parallel streams for any long operations (for example, network connections),  since all parallel streams work with one <code>ForkJoinPool</code>, such long operations can stop all parallel streams in the JVM due to  the lack of available threads in the pool, etc. e. parallel streams should be used only for short operations where the count  goes for milliseconds, but not for those where the count can go for seconds and minutes;</li>
<li>Saving order in parallel streams increases execution costs, and if order is not important, it is possible to disable its  saving and thereby increase productivity by using an intermediate operation <code>unordered()</code>:</li>
</ul>
